This page tests approaches to embedding an animated SVG, to verify which survive Mintlify's image optimizer on deployment.

## Approach A — Markdown image reference (BROKEN)

Imgix detects `width`/`height` on the SVG, generates srcset URLs with explicit `w=` params, and rasterizes to PNG/WebP — stripping SMIL animations.

![Animated SVG via markdown](/images/test-animation.svg)

---

## Approach B — Inline SVG markup (BROKEN)

MDX strips all SVG children during bundling — the `<svg>` tag renders empty.

<div>
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 200 200" width="200" height="200">
  <circle cx="100" cy="100" r="40" fill="#927746">
    <animate attributeName="r" values="40;70;40" dur="2s" repeatCount="indefinite" />
    <animate attributeName="opacity" values="1;0.3;1" dur="2s" repeatCount="indefinite" />
  </circle>
  <text x="100" y="170" textAnchor="middle" fontSize="14" fill="#e0d390">Pulsing circle</text>
</svg>
</div>

---

## Approach C — HTML `<img>` tag (BROKEN)

Mintlify also processes `<img>` tags through imgix — same rasterization as A.

<img src="/images/test-animation.svg" width="200" height="200" alt="Animated SVG via img tag" />

---

## Approach D — `<object>` tag (BROKEN)

Mintlify strips `<object>` tags entirely from the rendered output.

---

## Approach E — Markdown ref to SVG with viewBox only, no width/height (testing)

Root cause says imgix only processes SVGs with *known dimensions*. This SVG has only `viewBox`, no `width`/`height` attributes — imgix may skip srcset generation and serve it as-is.

![Animated SVG no size via markdown](/images/test-animation-nosize.svg)

---

## Approach F — `<iframe>` pointing to Mintlify-hosted SVG (testing)

An iframe creates a fully separate browsing context — but the SVG still goes through Mint CDN, which may sanitize it.

<iframe src="/images/test-animation.svg" width="200" height="200" style={{border:"none"}} />

---

## Approach G — `<iframe>` pointing to externally hosted SVG (testing)

SVG served directly from GitHub raw (outside Mintlify's CDN entirely). If the animation survives here, it confirms the CDN pipeline is the culprit and this is a viable customer workaround.

<iframe src="https://raw.githubusercontent.com/skipper4/documentation/main/images/test-animation.svg" width="200" height="200" style={{border:"none"}} />
